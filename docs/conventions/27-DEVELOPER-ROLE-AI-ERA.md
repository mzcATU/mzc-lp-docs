## 1. 개발자 역할 재정의

### 1.1 전통적 역할 → 새로운 역할

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        AI 시대의 개발자 역할 변화                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  [전통적 역할]                    [새로운 역할]                               │
│  ├── 코드 작성자                  ├── 방향 결정자 (Driver)                   │
│  ├── 버그 수정자                  ├── 승인자 (Approver)                      │
│  └── 기능 구현자                  ├── 검증자 (Verifier)                      │
│                                   ├── 코드 관리자 (Code Owner)               │
│                                   └── 서비스 딜리버리 담당자                  │
│                                                                              │
│  핵심 변화:                                                                   │
│  "개발자는 더 이상 단순한 소프트웨어 개발자에 머물러서는 안 된다.              │
│   코딩 행위를 넘어서 서비스를 딜리버리하는 사람으로서                          │
│   과업 전체를 바라보는 시각을 가져야 한다."                                   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 새로운 역할 상세

| 역할 | 설명 | 핵심 역량 |
|------|------|----------|
| **방향 결정자** | 무엇을, 왜, 어떻게 만들 것인지 결정 | 비즈니스 이해, 요구사항 분석 |
| **승인자** | AI 생성 결과물의 적합성 판단 | 코드 리뷰, 품질 기준 |
| **검증자** | 결과물이 요구사항을 충족하는지 확인 | 테스트, 검증 |
| **코드 관리자** | 코드베이스의 일관성과 품질 유지 | 아키텍처, 컨벤션 |
| **딜리버리 담당** | 개발 전 과정 책임 (1~6단계) | 요구사항→배포 전체 |

### 1.3 업무 이해의 중요성

```
핵심 인사이트:
"개발자가 방향 결정, 승인, 검증의 역할을 제대로 수행하기 위해서는
기술적인 문제보다 업무적인 문제로 이해하는 것이 필수적이다.
실제로 소프트웨어 개발에서 이 부분이 가장 어려운 부분이다."

개발자가 갖춰야 할 것:
├── 비즈니스 도메인 이해
├── 요구사항 분석 능력
├── 트랜잭션 바운더리 설계 능력
├── AI 결과물 검증 능력
└── 서비스 딜리버리 관점
```

### 1.4 사용자이자 생산자로서의 역할

```
개발자의 이중 역할:

1. 사용자로서:
   - 게시판, SNS 등 익숙한 서비스의 사용자
   - 서비스의 메커니즘과 구조를 파악하는 능력 보유

2. 생산자로서:
   - 단순 사용자 관점을 넘어서
   - 비즈니스 설계 단계에서 트랜잭션 바운더리 설계
   - AI 활용 시에도 명확한 방향 제시

예시:
- 게시물 삭제 시 댓글 처리 방법 → 기획자에게 질문하고 결정받아야 함
- 트랜잭션 단위를 묶는 것은 코드지만, 이유와 범위는 업무적 결정
```

## 2. 문서 이해 의무

### 2.1 핵심 원칙

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           문서 이해 의무                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  "문서의 작성 주체(AI로 만들었든, 직접 만들었든)와 관계없이,                    │
│   참여자들은 자신이 만든 문서를 어딘가에 공표하는 순간                          │
│   그 문서 안에 있는 모든 내용을 철저하게 이해해야 한다."                        │
│                                                                              │
│  이해하지 못하는 부분이 있다면:                                                │
│  ├── 해당 내용을 제외하거나                                                   │
│  └── 철저하게 공부해서 완전히 파악해야 함                                      │
│                                                                              │
│  목적:                                                                        │
│  ├── 문서의 신뢰성 확보                                                       │
│  └── 발표자/작성자의 전문성 확보                                              │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 실천 가이드

| 상황 | 해야 할 일 | 하지 말아야 할 일 |
|------|-----------|-----------------|
| AI가 문서 생성 | 내용 검토 후 이해 | 그대로 공유 |
| 이해 못하는 용어 | 학습 후 포함 또는 제외 | 모르는 채로 포함 |
| 문서 공유 전 | 모든 내용 설명 가능한지 확인 | 대충 훑어보고 공유 |
| 질문 받았을 때 | 자신있게 답변 | "AI가 만들어서 모르겠어요" |

### 2.3 체크리스트

```
문서 공유 전 확인:

□ 모든 용어의 의미를 설명할 수 있는가?
□ 문서의 구조와 흐름을 설명할 수 있는가?
□ 왜 이렇게 작성했는지 설명할 수 있는가?
□ 질문에 답변할 수 있는가?
□ 이해 못하는 부분은 제외했는가?
```

## 3. AI 활용 표준화

### 3.1 AI와의 대화 능력

```
"AI와 대화하는 것 또한 소프트웨어 개발자에게 필요한 중요한 능력이다."

필요한 역량:
├── 명확한 요구사항 전달
├── 컨텍스트 제공 능력
├── 결과물 검증 능력
└── 반복 개선 (Iteration) 능력
```

### 3.2 AI 제어 표준화

```
소프트웨어 개발과 관리를 효율적으로 진행하기 위해
AI를 어떻게 제어할지에 대한 표준화가 필요하다.

표준화 대상:
├── 슬래시 커맨드 (Slash Commands)
├── 프롬프트 셋 (Prompt Sets)
├── 컨벤션 로딩 방식
└── 결과물 검증 기준
```

### 3.3 AI 도구별 활용 가이드

| AI 도구 | 특성 | 활용 팁 |
|---------|------|---------|
| **Claude Code** | 코드 작성 선호, 설계 지시에도 코드 생성 | 슬래시 커맨드로 명확한 지시, `/plan` 활용 |
| **Codex** | 사용자 요구사항 준수, 필요한 코드만 작성 | 명확한 지시 기반 작업 |
| **Gemini** | 넓은 컨텍스트 이해 | 대규모 분석, 리뷰 |

### 3.4 컨텍스트 스위칭 주의

```
⚠️ AI 페어 코딩 환경의 주의점:

"AI 기반의 페어 코딩 환경에서 컨텍스트 스위칭이
최소화되는 것이 아니라, 오히려 엄청나게 빠른 속도로 극대화될 수 있다."

원인:
- AI의 작업 속도가 사람이 루틴 작업을 이해하고 처리하는 속도보다 훨씬 빠름
- 너무 많은 정보가 너무 빠르게 생성됨
- 개발자가 이를 따라가고 이해하는 데 인지적 부담 가중

대응:
├── 한 번에 하나의 작업에 집중
├── AI 결과물 검토 시간 확보
├── 단계별 승인 후 다음 진행
└── TodoWrite로 진행 상황 추적
```

## 4. 프롬프트 및 레퍼런스 관리

### 4.1 README.md 활용

```
README.md는 단순히 프로젝트를 설명하는 것을 넘어,
"어떤 것들을 중요하게 생각했는지"에 대한 내용을 기록하여
다음 프로젝트에 참고할 수 있는 레퍼런스로 활용해야 한다.

포함 내용:
├── 프로젝트 핵심 결정 사항
├── 아키텍처 선택 이유
├── 주요 트레이드오프
└── 배운 점 (Lessons Learned)
```

### 4.2 프롬프트 셋 관리

```
디버깅이나 코드 리뷰 시 유용한 프롬프트 셋을 별도로 관리한다.

예시 구조:
prompts/
├── code-review.md         # 코드 리뷰용 프롬프트
├── debugging.md           # 디버깅용 프롬프트
├── refactoring.md         # 리팩토링용 프롬프트
├── test-generation.md     # 테스트 생성용 프롬프트
└── documentation.md       # 문서화용 프롬프트
```

### 4.3 프롬프트 템플릿 예시

```markdown
## 코드 리뷰 프롬프트

다음 코드를 리뷰해주세요:

[코드 붙여넣기]

확인 사항:
1. 컨벤션 준수 여부 (참조: XX-CONVENTIONS.md)
2. 트랜잭션 바운더리 적절성
3. Race Condition 가능성
4. Edge Case 처리
5. 에러 핸들링
6. 보안 취약점

개선 제안이 있다면 구체적인 코드와 함께 제시해주세요.
```

### 4.4 슬래시 커맨드 활용

```
Claude Code 슬래시 커맨드 예시:

/plan          - 작업 계획 수립
/compact       - 컨텍스트 정리
/clear         - 대화 초기화
/doctor        - 시스템 상태 확인
/init          - 프로젝트 초기화
/review        - 코드 리뷰 요청

커스텀 커맨드 정의 가능:
- 프로젝트별 자주 사용하는 작업 정의
- 팀 내 표준화된 명령어 공유
```

## 5. 컨텍스트 제공

### 5.1 컨텍스트 정의

```
컨텍스트 = "목표를 달성하기 위해 필요한 모든 정보의 총체"

컨텍스트 리포지토리:
├── 프로젝트 개요
├── 아키텍처
├── 모듈 구조
├── 역할/권한
├── 트랜잭션 바운더리
├── Edge Case 정의
└── 기술 스택
```

### 5.2 정보 이해의 중요성

```
"구현하고자 하는 기능에 필요한 정보를 완전히 이해하지 못하면
컨텍스트를 놓치기 쉽다."

선행 조건:
├── 업무를 제대로 이해
├── 프로젝트의 방향을 인식
└── 필요한 정보 파악

→ 소프트웨어 개발의 모든 과정에서 선행되어야 함
```

### 5.3 AI에게 제공해야 할 핵심 컨텍스트

| 순위 | 항목 | 설명 |
|------|------|------|
| 1 | **데이터 (상태)** | 시스템의 근간, Entity 구조 |
| 2 | **API 스펙** | 무엇을 넣으면 무엇이 나오는지 |
| 3 | **트랜잭션 바운더리** | 작업 단위, ACID 원자성 |
| 4 | **권한** | 누가 무엇을 할 수 있는지 |
| 5 | **Edge Case** | 경계 조건, 제한 사항 |
| 6 | **에러 핸들링** | 실패 시 처리 방법 |
